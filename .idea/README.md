# v0n5ai IME

> **確率的エンベディング地図の成長を目指すプロジェクト**
>
> 入力から出力への写像が、確率分布として定義され、
> ユーザーとの相互作用を通じて成長する空間。

## 未来の入力とは何か

### 歴史の教訓

日本語入力の 40 年を振り返る：

- **1980s** - カナ打ちの時代、専門家のもの
- **1990s** - ローマ字普及、しかし「変換」という行為の強制
- **2000s** - 予測変換、Google の登場で「精度」競争へ
- **2010s** - 雲変換、プライバシーとのトレードオフ
- **2020s** - AI 変換、しかしまだ「単語→文」の延長線上

**根本的な問い**：なぜ人間は機械に合わせて「変換」をしなければならないのか？

### 構造的な不利

日本語入力は、英語・中国語と比べて**構造的に不利**である：

| 言語 | 入力 | 出力 | マッピング |
|------|------|------|------------|
| 英語 | `hello` | `hello` | 1:1 等価 |
| 中国語 | `nihao` | `你好` | 1:1 変換 |
| 日本語 | `konnichiwa` | `こんにちは` / `今日は` / `今にちは` | 1:N 確率 |

**問題の本質**：
- 英語：入力＝出力、思考がそのまま伝わる
- 中国語：ピンイン→漢字、一対一対応
- 日本語：ローマ字→ひらがな→漢字選択、**二段階の確率過程**

この不利を「変換精度」でカバーしようとするのは、間違ったアプローチだ。
**入力モデルそのもの**を再設計する必要がある。

### 現状への不満

既存 IME の問題点：

1. **モード選択の強制** - 英数/かな、ユーザーが切り替える
2. **離散的な変換** - 候補は「あるかないか」、確率ではない
3. **文脈の不在** - 直前の数単語しか見ていない
4. **誤字に不寛容** - 一文字違えば「該当なし」
5. **意味を理解していない** - 同音異義語の区別はユーザー任せ
6. **学習という名の妥協** - 辞書登録というアナクロニズム

### 他の技術からの応用

**ASR（自動音声認識）の知見**

音声認識は既に解いている：
- 曖昧な入力を確率分布で扱う
- 文脈全体で最適化
- 誤りを前提とした設計

**LLM の意味理解**

- 単語レベルではない、文の意味を捉える
- 同音異義語を文脈で自動選択
- 「変換」という行為そのものを不要に

**エディタの発想：CLI とプロンプトの切り替え**

現代のエディタ（VSCode, Zed）は：
- コマンドラインと編集画面をシームレスに統合
- **モードは文脈で決まる**、ユーザーは意識しない

この発想を IME に応用する：
- 「入力モード」と「確定モード」の境界を溶かす
- 機械が文脈を読んで、最適な出力形式を選択

**GUM（Grapheme-to-Unitem Mapping）の方向性**

- 入力文字列を直接、意味単位にマッピング
- 中間表現（ひらがな）を介さない
- エンドツーエンドの最適化

## 宣言

v0n5ai IME は以下を実現する：

### 1. 自動モード選択

```
入力： "mail send shito"  → 自動で英数混じり "mail send して"
入力： "今日は"           → 文脈で判断、そのまま漢字
入力： "konnnichiwa"     → 自動でひらがな/漢字最適化
```

**モード切り替えは機械が担当**。ユーザーは入力に集中するだけ。

### 2. 確率的入力

```
入力： "konnichiwa"
出力： "こんにちは" （確率 0.92）, "今日は" （確率 0.07）, ...
```

候補は「存在」ではなく「分布」で返る。ユーザーは最も確からしいものを「選ぶ」のではなく、システムが最も確からしいものを「提示」する。

### 3. 意味的変換

```
入力： "きょうはあめだ"
文脈： 天気の話 → "今日は雨だ"
文脈： 曜日の話 → "今日は金だ"
```

変換は意味で決まる。

### 4. 誤字許容

```
入力： "konnnichiwa"  → "こんにちは"
入力： "konnichiha"  → "こんにちは"
入力： "こんにちわ"  → "こんにちは"
```

誤りは「エラー」ではない。入力の揺れとして吸収する。

### 5. GUM アプローチ

```
入力文字列 → 意味ベクトル → 最適出力
（ひらがな中間表現を介さない）
```

### 6. ローマジ専用

かな入力の効率？物理キーボードという制約下での最適化は終わった。
私たちはローマ字入力という、すでに普及したインターフェースの上で、
**入力の本質**を再設計する。

- **GUM エンジン** - Grapheme-to-Unitem Mapping
